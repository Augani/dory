# Kubernetes Version Picker — Design Spec

**Track:** Research opportunity #4 (`docs/research/2026-07-02-dev-pain-points.md`) — close OrbStack's 2-year-open gap [orbstack#777](https://github.com/orbstack/orbstack/issues/777) (open since Nov 2023). OrbStack ships exactly ONE auto-bumped Kubernetes version and officially redirects users who need a specific version to `kind`/`k3d`. Dory runs k3s, which publishes every k8s minor as a tagged image on `rancher/k3s` — so a version picker at cluster-enable time directly converts the OrbStack users who are forced onto Kind today.

**What exists today:** `KubernetesProvisioner` (`Dory/Runtime/Kubernetes/KubernetesProvisioner.swift`) hardcodes `static let image = "rancher/k3s:v1.31.5-k3s1"`, runs that image as the `dory-k8s` container inside Dory's shared VM, publishes the API on `:6443`, waits for the node to become `Ready`, and writes `~/.kube/dory-config`. `AppStore.enableKubernetes()` (`Dory/Models/AppStore.swift`) is the one-click entry — gated on `runtimeKind == .sharedVM`, guarded by `kubernetesBusy`, driving a `progress` closure into `kubernetesInfo`. `KubernetesView.emptyState` (`Dory/Features/Tables/KubernetesView.swift`) renders the single "Enable Kubernetes" button (`accessibilityIdentifier("enable-kubernetes")`). `disableKubernetes()` tears the cluster down via `KubernetesProvisioner.disable`. The `readiness.sh --k8s` check (`test_k8s`) runs `KUBECONFIG=~/.kube/dory-config kubectl get nodes` — version-agnostic, so it must keep passing for whatever version is selected.

**What is missing:** the user cannot choose the k8s version. A team pinned to v1.32 for CI parity, or one that needs a v1.34 feature gate, has no path except abandoning Dory's one-click cluster for Kind — the exact OrbStack failure mode #777 documents.

**Goal:** let the user pick a k3s version at enable time — a small picker in the enable flow offering the latest patch of the last ~3 k8s minors (e.g. `v1.34`/`v1.33`/`v1.32`), defaulting to the newest. Persist the last choice so it becomes the default next time. Keep the existing enable/disable/kubeconfig machinery intact. `readiness.sh --k8s` must still pass.

## Access strategy decision (load-bearing): static curated version catalog, NOT a live query

The version list is a **static, curated, in-binary catalog** (`KubeVersionCatalog`) — a small ordered array of `KubeVersion` values, each pairing a display minor (e.g. `v1.34`) with the exact `rancher/k3s` tag it resolves to (e.g. `v1.34.1-k3s1`). It is NOT queried from the network at runtime. Justification:

- **Zero new network surface, zero new failure mode, works offline.** The enable flow already does exactly one network operation (`runtime.pull(image:)`) and tolerates its failure (`try?`). A live query against Docker Hub's `rancher/k3s` tags API (or the k3s channel server `https://update.k3s.io/v1-release/channels`) adds a second network dependency that can hang, rate-limit (Docker Hub anonymous pulls are throttled), or return an unparseable payload — and it would have to run *before* the picker can render, delaying or breaking the one UI the user sees. A curated list renders instantly and offline.
- **Every tag in the catalog is one we have chosen and can smoke-test.** k3s occasionally publishes tags that don't boot cleanly on Apple's virtualization, or `-rc`/`-alpha` tags a picker must not offer. A curated list is a reviewed allow-list; a live query would surface every tag including release candidates and require client-side filtering heuristics that rot.
- **Consistency with the codebase.** Dory already hardcodes the single k3s tag; nothing in the k8s stack performs registry discovery. The catalog is the minimal, in-keeping generalization of the existing constant from one value to ~3.
- **Cheap to keep current.** The catalog lives in one file with a dated `lastReviewed` comment; bumping it to the next minor is a one-line PR, the same cadence at which the hardcoded constant was already being bumped. The staleness cost is bounded (a user on the newest catalog entry is at most one minor behind upstream until the next release) and is strictly better than today's single frozen version.

The durable cost is that a brand-new k8s minor is not offered until the catalog is bumped. This is acceptable: it is the identical maintenance the single constant already required, now yielding 3 choices instead of 1, and it is the trade OrbStack itself effectively makes (they ship one curated version). A pure `KubeVersionCatalog` seam keeps the door open to a live-query source later without touching the provisioner or the view — the provisioner consumes a resolved tag string, not the catalog.

## Decisions

- **`KubeVersionCatalog` (`Dory/Runtime/Kubernetes/KubeVersionCatalog.swift`, new, pure/`Sendable`).** Owns:
  - `struct KubeVersion: Identifiable, Hashable, Sendable { let minor: String; let tag: String }` where `minor` is the human label (`"v1.34"`) and `tag` is the full `rancher/k3s` image tag (`"v1.34.1-k3s1"`). `id` is `tag`. `image` computed → `"rancher/k3s:\(tag)"`.
  - `static let all: [KubeVersion]` — the curated list, newest first (index 0 = default). Seeded with the last three GA minors as `rancher/k3s` tags; carries a `// lastReviewed: 2026-07-02` marker.
  - `static var latest: KubeVersion { all[0] }` (the array is a compile-time non-empty literal; `all[0]` is safe by construction and asserted by a test).
  - `static func version(forTag tag: String?) -> KubeVersion` — resolves a persisted tag back to a catalog entry, falling back to `latest` when the tag is `nil` or no longer in the catalog (so a removed/renamed tag degrades to newest instead of a dangling selection).
- **`KubernetesProvisioner` is parameterized by tag, not by a global constant.** Replace `static let image = "rancher/k3s:v1.31.5-k3s1"` with `static let defaultImage = KubeVersionCatalog.latest.image` (preserving a sensible default for any call site that omits a version) and thread an `image: String` parameter through:
  - `enable(runtime:image:progress:)` — `image` defaults to `defaultImage`; `createBody(image:)` interpolates the passed tag instead of the constant. Every other line (pull, create, start, Ready-wait, kubeconfig write) is unchanged.
  - `isRunning`, `disable`, `deleteExisting`, `writeKubeconfig` are version-independent (they key off `containerName`/`kubeconfigPath`) and are untouched.
- **Selection state + persistence live in `AppStore`.** Add `var kubernetesVersionTag: String` initialized from `UserDefaults` (`Self.kubernetesVersionKey = "dory.kubernetesVersion"`) resolved through `KubeVersionCatalog.version(forTag:)` (so a stale persisted tag falls back to latest). A `setKubernetesVersion(_ version: KubeVersion)` setter updates the property and writes the tag to `UserDefaults`, mirroring the existing `setAutoUpdate`/`setRouteDockerCLI`/`setAppearance` persistence pattern (all `Self.<name>Key` + `UserDefaults.standard.set`). Under the `realLaunch` gate in `init` (matching every other persisted setting), the saved tag is loaded; automation/test/snapshot launches keep the default.
- **`enableKubernetes()` passes the selected tag.** `KubernetesProvisioner.enable(runtime: runtime, image: KubeVersionCatalog.version(forTag: kubernetesVersionTag).image) { … }`. The `runtimeKind == .sharedVM` gate, `kubernetesBusy` guard, `progress`→`kubernetesInfo` wiring, and the trailing `await loadKubernetes()` are all unchanged.
- **Re-enable with a different version = disable + recreate (explicit, confirmed).** k3s cannot change its server version in place; a version switch is a destructive cluster recreate (workloads are lost). The provisioner's `enable` already calls `deleteExisting` before create, so a straight re-enable at a new tag *would* recreate — but silently discarding a running cluster's workloads is unacceptable. Decision: when a cluster is **already running** and the user picks a **different** version, the change is gated behind a `confirmationDialog` ("Switching to <minor> recreates the cluster. All running workloads will be lost. This cannot be undone.") that, on confirm, runs `disableKubernetes()` then `enableKubernetes()`. Picking the **same** version, or picking any version while **no cluster is running**, needs no confirm — it just sets the default and enables. This mirrors the WS5/actions-parity rule: no destructive action without a confirm.
- **UI: a compact version `Picker` in the enable flow, defaulting to newest.**
  - **Empty state (`KubernetesView.emptyState`)**: add a `Picker` above the "Enable Kubernetes" button, bound to `store.kubernetesVersionTag`, options from `KubeVersionCatalog.all` (label = `minor`), `accessibilityIdentifier("kube-version-picker")`. It is disabled while `kubernetesBusy` and when `runtimeKind != .sharedVM` (same gate as the button), styled `.menu` and `.fixedSize()` to stay compact. Selecting an option calls `store.setKubernetesVersion(_:)` — this only sets the persisted default; it does not enable anything until the button is pressed. The button label gains the chosen minor for clarity (e.g. "Enable Kubernetes v1.34").
  - **Running cluster (`KubernetesView.banner` `⋯` menu)**: add a "Kubernetes Version" submenu (a `Menu` inside the existing `Menu`) listing `KubeVersionCatalog.all`, with a checkmark on the currently-running/selected minor. Choosing a *different* entry sets `pendingVersionSwitch` (a new `@State`) which drives the recreate `confirmationDialog`; choosing the current entry is a no-op. This is the "change version on an existing cluster" path and is the one that recreates.
- **Which version is "running" is derived, not guessed.** After enable, `loadKubernetes()` already surfaces the live server version in `kubernetesInfo` via `KubernetesProvider.status()` (the mock path shows `"v1.31.0 · …"`). The banner submenu's checkmark keys off `store.kubernetesVersionTag` (the selection Dory enabled with) — the source of truth for "what did we start", independent of the parsed display string. No new cluster query is introduced.
- **`readiness --k8s` is unaffected by design.** `test_k8s` runs `kubectl get nodes` against `~/.kube/dory-config` — it asserts the cluster is reachable, not that it is any particular version. Every catalog tag is a real GA k3s image that boots to a Ready single node and writes the same `k3s.yaml`, so the check passes for any selected version. No change to `readiness.sh` is required or made.

## Components

- **`KubeVersionCatalog`** (`Dory/Runtime/Kubernetes/KubeVersionCatalog.swift`, new, pure/testable): `KubeVersion` struct + `all` (curated, newest-first) + `latest` + `version(forTag:)`. The single place tags are declared; the unit-test target for default/fallback resolution.
- **`KubernetesProvisioner`** (`Dory/Runtime/Kubernetes/KubernetesProvisioner.swift`, modify): `static let defaultImage` replaces the hardcoded constant; `enable(runtime:image:progress:)` + `createBody(image:)` take a tag. No behavior change when `image` is omitted (defaults to latest).
- **`AppStore`** (`Dory/Models/AppStore.swift`, modify): `var kubernetesVersionTag`, `Self.kubernetesVersionKey`, `init` load under `realLaunch`, `setKubernetesVersion(_:)` persist, `enableKubernetes()` passes the resolved image, and a `switchKubernetesVersion(_:)` async helper (disable→enable) for the confirmed recreate path.
- **`KubernetesView`** (`Dory/Features/Tables/KubernetesView.swift`, modify): version `Picker` in `emptyState`; "Kubernetes Version" submenu + recreate `confirmationDialog` (`pendingVersionSwitch: KubeVersion?`) in the running-cluster surface; enable-button label reflects the chosen minor.

## Error handling

- `KubeVersionCatalog.all` is a compile-time non-empty literal; `latest` therefore never traps. `version(forTag:)` never returns nil — an unknown/`nil` tag resolves to `latest`, so a persisted tag that was dropped from a future catalog degrades gracefully to newest rather than dangling or crashing.
- Persistence uses the same defensive read as every other setting: `UserDefaults.standard.string(forKey:)` is optional-chained and passed through `version(forTag:)`, so a missing or garbage default value yields `latest`.
- The provisioner's failure surfaces are unchanged: `enable` still throws `K8sError.createFailed`/`.notReady`/`.kubeconfigFailed`, `enableKubernetes()` still catches into `kubernetesInfo = "Kubernetes failed to start"`. A version that fails to pull/boot fails exactly as the single hardcoded version does today — no new error path.
- The recreate path routes through the existing `disableKubernetes()`/`enableKubernetes()` (both `kubernetesBusy`-guarded), so a mid-switch second trigger is ignored; failure of the enable half lands in the existing `kubernetesInfo` failure text. The destructive switch is never reached without the `confirmationDialog` confirm.

## Testing

Pure logic is unit-tested in `DoryTests/` (no cluster required), mirroring `KubeRowMapperTests`/`KubeResourceKindTests`:

- **`KubeVersionCatalogTests`** — `all` is non-empty and newest-first (index 0 == `latest`); every `tag` is a well-formed `rancher/k3s` k3s tag (`hasPrefix("v")` and `contains("-k3s")`); every `image` is `"rancher/k3s:\(tag)"`; `version(forTag:)` returns the matching entry for a known tag, returns `latest` for `nil`, and returns `latest` for an unknown tag; minors are distinct.
- **`KubernetesProvisionerImageTests`** — `KubernetesProvisioner.defaultImage == KubeVersionCatalog.latest.image`; `createBody(image:)` for a given tag interpolates `"rancher/k3s:<tag>"` into the `Image` field and is otherwise byte-identical in structure to the default (assert the JSON contains `"Image":"rancher/k3s:v1.32.…-k3s1"` and still contains the `server`/`--disable=traefik`/`PortBindings` scaffold). `createBody` is exposed at the needed access level for the test (or a pure static `createJSON(image:) -> String` is extracted and tested).
- **`AppStoreKubernetesVersionTests`** (`@MainActor`, mirrors `KubeResourceLoadStateTests`) — a fresh `AppStore` defaults `kubernetesVersionTag` to `KubeVersionCatalog.latest.tag`; `setKubernetesVersion(_:)` updates the property and the resolved `version(forTag:)` round-trips to the chosen entry; setting an entry and reading back through `version(forTag:)` yields that entry's image.

**Build + snapshot verification** for the SwiftUI surface (not unit-testable): `scripts/build.sh` gates compilation (`BUILD SUCCEEDED`, `xcodebuild_exit=0`, objectVersion stays 77 — never open Xcode GUI). `scripts/shots.sh` captures `KubernetesView` empty state (via the mock backend, `DORY_SECTION=kubernetes`) showing the version picker above the enable button. New files auto-compile via `fileSystemSynchronizedGroups` — no `.pbxproj` edit. `readiness.sh --k8s` (`RUN_K8S=1`) remains green: `kubectl get nodes` against `~/.kube/dory-config` is version-agnostic.

## Non-goals (this cycle)

- **Live version discovery** (Docker Hub tags API / k3s channel server) — the curated static catalog is the deliberate source (see access strategy); the `KubeVersionCatalog` seam leaves a live source swappable later.
- **In-place cluster upgrade** — k3s server versions cannot change without recreate; the version switch is an explicit, confirmed destroy+recreate. No node-drain/backup-restore of workloads across the switch.
- **Arbitrary/custom tag entry** — the picker offers only the curated allow-list (no free-text `rancher/k3s:<anything>`); this bounds the tested surface. A power-user escape hatch is out of scope.
- **Multiple concurrent clusters at different versions / multi-node** — Dory provisions the single `dory-k8s` cluster; the picker chooses its version, not a fleet.
- **Distro choice (k3s vs k0s/kind/vanilla)** — Dory's cluster is k3s; this spec picks the k3s *version*, not the distribution.
- **Auto-bumping the catalog from CI** — the catalog is bumped by a reviewed PR with a `lastReviewed` date, the same cadence the single constant already used.
